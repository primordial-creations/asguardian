Research Report: Automated Static Detection of Hexagonal Architecture ViolationsIntroduction to Architectural Conformance VerificationThe paradigm of Hexagonal Architecture, originally codified as "Ports and Adapters" by Alistair Cockburn in 2005, establishes a strict isolation protocol between an application's core business logic (the domain) and its external technical integrations (infrastructure, user interfaces, and databases). The primary architectural directive demands that dependencies flow exclusively inward; the core domain must remain entirely agnostic of the infrastructure that surrounds it. To interact with the external world, the domain defines abstract "ports" (interfaces), which are subsequently implemented by technology-specific "adapters" residing in the outer infrastructure layers.While this separation theoretically maximizes testability, delays technological lock-in, and enforces the Single Responsibility Principle, maintaining this structural purity over a software system's lifecycle presents a formidable challenge. As codebases evolve, developers often inadvertently introduce topological faults—such as the domain layer directly instantiating a database repository or a user interface controller bypassing the application use-case layer to query the database. This phenomenon, known as architectural drift or architectural erosion, silently compromises the system's maintainability until the boundaries become indistinguishable, resulting in a system often described as a "big ball of mud".To combat architectural erosion, the software engineering discipline has shifted from manual code reviews and informal conventions to automated static architecture-conformance checking. The objective of these automated systems is to verify that the "as-implemented" physical architecture mathematically aligns with the "as-intended" conceptual architecture without requiring the code to be executed dynamically.This report provides an exhaustive analysis of the formal models, heuristic mechanisms, and state-of-the-art tooling utilized to statically detect Hexagonal Architecture violations. By dissecting ecosystem-specific tools—such as ArchUnit for Java, dependency-cruiser for JavaScript/TypeScript, Deptrac for PHP, and NDepend for.NET—as well as the emerging cross-language capabilities of Tree-sitter, this document outlines the exact signals that are detectable without runtime execution. Furthermore, it addresses the fundamental limitations of simplistic lexical scanning and charts the industry transition toward deep semantic graphing.The Paradigm Shift: From Regex Lexical Scanning to Semantic AST AnalysisFor many engineering teams, initial attempts to enforce architectural boundaries rely on basic text-scanning tools or Regular Expression (Regex) matching within Continuous Integration (CI) pipelines. These primitive checkers typically scan source files for prohibited string patterns, such as searching for the string import java.sql.* or require('express') within files located in a domain/ directory. While trivial to implement, lexical scanning is inherently brittle and fundamentally inadequate for rigorous architectural conformance.The inadequacy of Regex stems from its lack of semantic awareness. Lexical scanners operate on raw character strings and possess no understanding of the language's grammar, scope, or execution context. Consequently, they are easily subverted by modern programming idioms. Aliased imports (e.g., import { UserRepository as Repo } from './infrastructure') effectively mask the true nature of a dependency from a simple string matcher. Furthermore, multiline import declarations, dynamic module loading, and the presence of prohibited keywords within benign contexts (such as block comments or string literals) result in an unacceptable rate of false positives and false negatives. More critically, Regex cannot comprehend inheritance hierarchies or interface realization. It cannot determine if a class implements a specific interface, making it impossible to verify the foundational Ports and Adapters contract where an infrastructure component must fulfill a domain-defined port.To accurately detect violations, analysis must transition from the lexical phase to the syntactic and semantic phases of compiler theory. This requires parsing the source code into an Abstract Syntax Tree (AST) or generating a Concrete Syntax Tree (CST). An AST is a hierarchical tree representation of the abstract syntactic structure of source code, where each node denotes a construct occurring in the code (such as variable declarations, method invocations, and class definitions).By operating at the AST level, static analysis tools construct a high-fidelity directed graph of the software's true topological dependencies. This semantic graph allows tools to query structural relationships programmatically. Instead of asking, "Does this file contain the word 'database'?", an AST-aware tool asks, "Does this ClassDeclaration node, which resides within the conceptual Domain layer, contain an ObjectCreationExpression node that instantiates a class mapped to the Infrastructure layer?" This shift from lexical pattern matching to semantic graph traversal forms the bedrock of modern architectural enforcement.Formal Models and Academic Perspectives on Architectural ConformanceThe detection of architectural violations via static analysis is deeply grounded in formal models that mathematically represent software structure. Academic research has long pursued methods to automate architectural conformance checking, striving to bridge the gap between abstract architectural intent and concrete physical implementation.Reflexion Models and Dependency-Structure MatricesTwo foundational paradigms dominate the academic literature regarding static conformance checking and structural analysis: Reflexion Models (RMs) and Dependency-Structure Matrices (DSMs).The Reflexion Model technique, pioneered in software engineering research, requires the architect to define a high-level conceptual model of the desired system. In the context of Hexagonal Architecture, this conceptual model would consist of strictly defined nodes representing the Domain, Application, and Infrastructure layers, along with directed edges representing the allowed inward dependencies. A specialized extraction tool then parses the codebase to build a physical source model, which represents the actual dependency graph extracted from ASTs or bytecode. The engineer provides a declarative mapping linking the physical source artifacts (directories, namespaces, classes) to the conceptual entities. The engine then computes a "reflexion" that compares the two models, categorizing the physical dependencies into three distinct states. Convergences represent implemented dependencies that perfectly align with the allowed conceptual rules. Divergences highlight prohibited dependencies, such as an implementation where the Domain layer illegally references the Infrastructure layer, triggering a structural violation flag. Finally, absences indicate expected dependencies defined in the conceptual model that have not been implemented in the source code, which can be useful for identifying missing features or unfulfilled interface contracts.Conversely, Dependency-Structure Matrices (DSMs) offer a purely mathematical and visual representation of system coupling. A DSM is an adjacency matrix representing the components of a system, where both the rows and columns denote classes, packages, or architectural layers. A mark or numeric value in a specific matrix cell indicates that the element in the corresponding row depends on the element in the corresponding column. By algorithmically clustering and organizing the rows and columns into hierarchical layers, architectural violations become immediately visible as marks appearing out of sequence—typically above or below the matrix diagonal, depending on the chosen convention. This matrix representation is particularly powerful because it allows for the application of established linear algebra and graph theory algorithms to detect cyclical dependencies and calculate structural cohesion metrics without relying on complex graph visualization rendering.Formal Model ConceptPrimary MechanismAdvantage in Hexagonal AnalysisReflexion Models (RMs)Mapping physical source graphs to conceptual architectural diagrams.Explicitly highlights divergences (violations) where the Domain accesses outer layers, and absences where Ports are not implemented.Dependency-Structure Matrices (DSMs)Adjacency matrices representing module dependencies.Enables algorithmic cycle detection and provides a clear mathematical visualization of layer boundaries and unauthorized crossings.The Threat of AI-Induced Architectural ErosionRecent academic literature has highlighted an entirely new vector for rapid architectural erosion: Large Language Models (LLMs) and autonomous AI coding agents. As these tools transition from simple autocomplete utilities to complex system architects, their impact on long-term structural maintainability has become a subject of intense academic scrutiny. A pivotal 2025 pilot study introduced an empirical framework to measure "Architectural Erosion" and the accumulation of technical debt in AI-synthesized microservices.In this study, researchers tasked several state-of-the-art models (including GPT-5.1, Claude 4.5 Sonnet, and Llama 3 8B) with implementing a standardized business service under strict Hexagonal Architecture constraints. Utilizing deep AST parsing to evaluate the generated code, the researchers uncovered a stark divergence in architectural conformance capabilities across different model classes. While proprietary frontier models demonstrated exceptionally high adherence—with GPT-5.1 achieving a 0% violation rate—smaller, open-weights models exhibited catastrophic structural failures.Specifically, the Llama 3 model demonstrated an 80% Architectural Violation Rate. The predominant failure mode involved the LLM completely bypassing the interface adapters (Ports), instead creating illegal circular dependencies by directly wiring the Domain layer to the Infrastructure persistence components. Furthermore, the researchers identified a novel phenomenon termed "Implementation Laziness." To conserve token generation and satisfy context window constraints, open-weights models frequently omitted complex business logic from the domain core, resulting in anemic domain models that relied heavily on external frameworks to function.This research is supported by concurrent frameworks such as the "Theory of Code Space" (ToCS), which benchmarks an AI agent's ability to maintain coherent architectural beliefs during codebase exploration under partial observability. The ToCS framework evaluates whether agents can discover and respect cross-cutting architectural invariants and design intent. The findings from these academic benchmarks underscore a critical industry reality: as the deployment of automated code generation accelerates, the necessity for rigorous, automated architectural linting using ASTs and formal models is no longer optional. Without static enforcement, AI-generated code will rapidly accelerate the accumulation of structural technical debt, collapsing the boundaries of Hexagonal architectures.Detectable Violation Patterns at the AST and Import LevelTo automate the detection of Hexagonal Architecture violations, the abstract principles of the pattern must be translated into explicit, graph-queryable structural signatures. When source code is parsed into an Abstract Syntax Tree (AST) or when compiled code is analyzed as a bytecode dependency graph, several distinct topological faults become statically detectable without requiring runtime execution. These patterns represent the core checks implemented by all enterprise-grade architectural linting tools.1. The Domain Layer Importing Infrastructure PackagesThe most severe and fundamental violation of Hexagonal Architecture occurs when the innermost ring, the Domain layer, establishes a direct dependency on an outer layer, typically Infrastructure or Presentation. The defining characteristic of a pure domain layer is its technology agnosticism; it must remain completely isolated from databases, web frameworks, external APIs, and even standard library networking protocols. This isolation guarantees that business rules can be tested in a vacuum and that underlying technologies can be swapped without rewriting core logic.At the AST level, this violation is highly visible. The analysis tool queries for ImportDeclaration or RequireStatement nodes originating within any file classified as belonging to the Domain. It then extracts the string literal denoting the target of the import. If this target resolves to a module, package, or namespace that has been classified as Infrastructure—such as an Object-Relational Mapping (ORM) library, an HTTP client, or a specific user interface component—a violation is flagged.In compiled languages analyzed via bytecode, the signal is slightly different but equally explicit. A bytecode analysis engine, utilizing a visitor pattern over the compiled .class files, will examine the constant pool and method signatures of Domain classes. If a ClassVisitor detects a method parameter, a return type, a field declaration, or an invokevirtual instruction that references a fully qualified class name belonging to an external framework or database package, the structural boundary has been breached.2. The Adapter Failing to Implement a Port InterfaceHexagonal Architecture relies heavily on the Dependency Inversion Principle, implemented via the "Ports and Adapters" pattern. The Domain exposes abstract "Ports" (typically language-level interfaces) that define contracts for the capabilities it requires, such as data persistence or external messaging. The Infrastructure layer provides "Adapters" (concrete classes) that implement these specific interfaces. A critical architectural failure occurs when an infrastructure component interacts with the domain without fulfilling an established contract, or when a piece of infrastructure logic exists but is never bound to a domain port, rendering it inaccessible or creating rogue bypasses.Detecting this requires semantic analysis capable of understanding object-oriented inheritance and interface realization. At the AST level, the analyzer inspects ClassDeclaration nodes that are heuristically or explicitly mapped to the Infrastructure layer. The tool queries these nodes to verify the presence of an implements (or equivalent) syntactic clause. Furthermore, the tool must resolve the target of that implementation clause to ensure the referenced interface resides within the Domain or Application layer. Advanced AST querying systems, such as Tree-sitter, can utilize complex S-expressions to search specifically for class declarations that lack interface realization nodes targeting the domain, effectively highlighting "missing ports" and rogue adapters.3. Application Use Cases Depending on Concrete AdaptersThe Application layer, residing just outside the pure Domain, is responsible for housing Use Cases (sometimes called Interactors or Application Services). These Use Cases orchestrate business logic by coordinating Domain entities and interacting with external systems exclusively through abstract ports. A common violation occurs when a Use Case directly instantiates or references a concrete infrastructure class (e.g., executing new SqlUserRepository()) rather than accepting an interface via dependency injection (e.g., demanding an IUserRepository through its constructor). This couples the Use Case to a specific technology, destroying the pluggable nature of the architecture.The AST signal for this violation is the presence of instantiation syntax directed at concrete infrastructure. The analysis engine queries the bodies of Use Case classes for NewExpression or ObjectCreationExpression nodes. If the type being instantiated resolves to a class defined in the Infrastructure layer, a violation is recorded. Similarly, direct static method calls targeting concrete infrastructure utilities are detected by analyzing CallExpression nodes. In a bytecode environment, this is detected by observing an invokespecial instruction (which is utilized for constructor calls) targeting an infrastructure class from within a method body belonging to an application layer class.4. Circular Dependencies and Bidirectional CouplingBidirectional or cyclical dependencies utterly undermine the hierarchical isolation required by strict architectural layering. For example, if a Presentation controller reaches into the Data Access layer, and a utility within the Data Access layer subsequently invokes a formatting service located in the Presentation layer, the architectural boundary collapses into a tightly coupled loop.While detecting direct imports is relatively straightforward, identifying multi-step cyclical dependencies requires robust graph analysis. Once the analysis tool extracts all module connections from the ASTs or bytecode into a directed graph, it executes graph traversal algorithms. Algorithms such as Tarjan's strongly connected components algorithm or simple depth-first search cycle detection are employed to identify loops that cross predefined layer boundaries. If a path can be traced from Layer A to Layer B and back to Layer A, the tool flags a severe structural violation, demanding that developers break the cycle, typically by introducing a new shared abstraction or leveraging dependency inversion.5. Anemic Domain Models and Infrastructure LeakageWhile slightly more abstract than direct import violations, advanced static analysis engines can identify architectural antipatterns such as the "anemic domain model." In Domain-Driven Design, a core tenet of Hexagonal Architecture, domain entities must encapsulate both data and the rich business behaviors (methods) that operate on that data. An anemic domain model consists of entities that are merely data structures—possessing field declarations and basic getter/setter methods, but entirely lacking substantive business logic.Static analysis can flag this by evaluating the AST of Domain classes. If the ClassDeclaration contains only PropertyDeclaration nodes and trivial accessors, without any complex MethodDeclaration nodes, it triggers an anemic model warning. Furthermore, a critical infrastructure leak occurs if framework-specific metadata appears within the domain. While explicit, manual configuration is possible and often recommended, it is tedious to maintain at scale. Consequently, modern tools increasingly rely on structural heuristics, naming conventions, and topological signatures to autonomously identify architectural layers, particularly when retrofitting analysis onto legacy or unconfigured codebases.Directory and Namespace Path MatchingThe most universally applied heuristic relies on the physical organization of the file system and namespace conventions. Modern software development frameworks often encourage segregating components into directories that conceptually correspond to their architectural purpose.Tools exploit this by mapping physical paths to logical layers using Regular Expressions (Regex) or glob patterns. For instance, files residing within directories named src/domain/, src/core/, or utilizing namespaces such as App\Domain are automatically classified as belonging to the innermost core layer. Conversely, files located within paths like src/infrastructure/, src/adapters/, src/presentation/, or src/api/ are confidently mapped to the outer peripheral layers. By combining these path rules with the extracted AST dependency graph, tools can rapidly determine if an import statement crosses an illegal boundary based simply on the origin and destination directory strings.Class Name Suffixes and SignaturesIn codebases where directory structures are heterogeneous, fragmented, or poorly maintained, class naming conventions serve as powerful categorization heuristics. The software engineering community has largely standardized around specific suffixes that strongly imply an object's architectural role.Repositories: Classes suffixed with *Repository (e.g., UserRepository, OrderRepository) are universally recognized as Driven Adapters. Following patterns documented by Martin Fowler, a repository mediates between the domain and data mapping layers, acting as an in-memory collection. Static analysis tools definitively classify these objects as residing in the Infrastructure persistence layer.Controllers and Presenters: Classes ending in *Controller, *Presenter, or *Resolver signify Driving Adapters. These components are responsible for handling HTTP requests, GraphQL queries, or GUI events, clearly placing them in the outer Presentation layer.Services and Use Cases: Classes named *Service, *UseCase, *Handler, or *Command typically represent Application layer orchestrators. They form the middle layer, bridging the presentation mechanisms with the core domain entities.Topological and Structural Signals in Legacy CodebasesWhen analyzing deeply entangled legacy codebases where both directory structures and naming conventions have broken down, advanced analysis tools must rely on the pure mathematical shape of the dependency graph itself to infer layers.One primary topological signal is the calculation of Fan-In and Fan-Out ratios. "Fan-in" measures the number of incoming dependencies a module has, while "Fan-out" measures its outgoing dependencies. A component exhibiting exceptionally high fan-in from diverse areas of the codebase, coupled with very low fan-out, is typically identified as a foundational element, such as a core Domain entity or a widely used value object. Conversely, a module with high fan-out but zero fan-in—meaning it orchestrates many components but is called by nothing—is likely an application entry point or a high-level UI controller. However, if a module exhibits both massive fan-in and massive fan-out, it is flagged as a structural anomaly, often referred to as a "God module," which indicates a collapsed architectural boundary that requires immediate refactoring.Furthermore, interface implementation topologies provide strong signals. If static analysis detects that a specific class implements an interface, and that interface is heavily utilized by classes deemed to be core business logic, the implementing class can be heuristically identified as an Infrastructure Adapter, regardless of its physical location or name. Similarly, the external dependency footprint acts as an anchor; any class that imports heavily from external, third-party libraries (such as database drivers, message broker SDKs, or cloud provider APIs) is confidently flagged as belonging to the outer Infrastructure layer, as domain core classes should theoretically possess zero external dependencies.Tool Analysis 1: The Java Ecosystem (ArchUnit)Within the Java ecosystem, ArchUnit represents the definitive standard for enforcing architectural integrity. ArchUnit allows developers to define complex architectural constraints as standard unit tests, leveraging a highly expressive Fluent API to turn architectural documents into executable code.Fluent API and Rule DefinitionArchUnit's design philosophy treats architecture verification not as a separate build step, but as a standard testing procedure seamlessly integrated with frameworks like JUnit. Rules are constructed using a chained, domain-specific language that closely mimics natural language, making the tests highly readable.To enforce a strict Hexagonal Architecture boundary, a developer utilizes this API to explicitly isolate the domain. A foundational rule might be written as:
classes().that().resideInAPackage("..domain..").should().onlyDependOnClassesThat().resideInAnyPackage("..domain..", "java..").
This code explicitly declares that any class within the domain package is forbidden from depending on anything outside of the domain itself, with an exception carved out for standard Java base libraries. Similarly, the framework can mathematically enforce the Ports and Adapters contract by requiring that adapters actually implement their intended interfaces: classes().that().haveNameMatching(".*Adapter").should().implement(Port.class). These fluent rules catch unintended imports the moment a developer runs their local test suite, preventing architectural drift before code is committed.Under the Hood: ASM Bytecode AnalysisA critical architectural distinction in ArchUnit's methodology is that it does not parse Java source files or construct an Abstract Syntax Tree (AST). Instead, it operates entirely on compiled Java bytecode (.class files).Under the hood, ArchUnit utilizes the ASM bytecode manipulation framework, a highly optimized library used universally across the JVM ecosystem for dynamic class generation and deep bytecode analysis. The analysis process begins with ArchUnit's ClassFileImporter, which locates compiled files from the classpath and feeds them into ASM's event-based API. As ASM streams the raw bytecode, it triggers precise callbacks on a ClassVisitor.The ClassVisitor acts as the detection engine. The visitField callback extracts field types, establishing structural data dependencies between classes. The visitMethod callback captures method signatures, exposing dependencies passed as parameters or defined as return types. Crucially, to detect behavioral dependencies such as method calls and object instantiations nested deep inside a method body, ArchUnit relies on ASM's MethodVisitor. This visitor tracks individual bytecode instructions, looking specifically for invokespecial (which indicates constructor calls and object creation), invokevirtual (indicating instance method invocations), and getfield (indicating state access). By aggregating these granular bytecode events, ArchUnit constructs a massive, perfectly accurate dependency graph of the compiled application.Limitations and Capabilities of Bytecode AnalysisOperating at the bytecode level provides ArchUnit with immense execution speed and reliability. It seamlessly traces dependencies into pre-compiled third-party JARs, ensuring that even external libraries do not violate structural rules. Furthermore, it integrates tightly with JUnit 5 via the ArchUnitRunner and specialized extension APIs, which aggressively cache the heavy JavaClasses object graph across multiple test runs to minimize execution overhead during local development.However, bytecode analysis carries inherent limitations. Because the Java compiler aggressively optimizes code and erases certain semantic structures during compilation, ArchUnit loses access to purely source-level syntactic constructs. The most prominent example is type erasure, where generic type parameters are removed at runtime, making deep generic analysis challenging. Additionally, if the code is compiled without debugging symbols, local variable names are stripped, slightly obscuring the context of internal method dependencies. Despite these minor blind spots, bytecode analysis remains the most robust method for Java architecture validation.Tool Analysis 2: The JavaScript/TypeScript Ecosystem (Dependency-Cruiser)For ecosystems utilizing JavaScript and TypeScript, static analysis presents unique challenges. Dynamic typing, complex module bundlers, varied import syntaxes (CommonJS vs. ES Modules), and the frequent use of dynamic imports complicate structural evaluation. To solve this, dependency-cruiser provides a highly specialized, configuration-driven solution for architectural validation tailored specifically to the nuances of JS/TS.Import Graph Analysis over Deep AST ParsingUnlike tools that perform deep AST parsing to evaluate internal class structures, field access, and internal method calls, dependency-cruiser operates primarily as a high-performance import graph analyzer. It recursively parses files to identify import, require, and export statements, utilizing these connections to construct a comprehensive directed graph of module dependencies. It is sophisticated enough to handle both static JavaScript/TypeScript imports and dynamic imports, provided the dynamic import utilizes a static string argument or a template expression devoid of complex variables. This capability ensures that modern asynchronous module loading techniques, such as Webpack code splitting, do not hide architectural violations.Configuration: Regex Paths, Layers, and RulesetsBecause JavaScript lacks the strict namespace conventions of languages like Java or C#, dependency-cruiser does not automatically infer architectural layers. Instead, the tool requires developers to explicitly define the architecture using Regular Expressions applied to file paths within a configuration file (typically .dependency-cruiser.js or JSON).The configuration architecture is structured around three primary rule arrays: forbidden, allowed, and required.Forbidden Rules: These are the primary mechanism for enforcing the isolation of the Hexagonal core. A developer configures a rule where the from attribute specifies a regex matching the core (e.g., ^src/domain/) and the to attribute specifies a regex matching the outer layers (e.g., ^src/infrastructure/). When dependency-cruiser builds the import graph, any detected connection bridging these mapped paths triggers a critical violation.Required Rules: While forbidden rules prevent bad connections, required rules enforce structural contracts. For instance, an architecture might demand that every specific UI controller extends a base class. A required rule can specify that any file matching a controllers directory path must possess a direct dependency on a base-controller file.Advanced Topological Detection: Beyond simple path-to-path string matching, dependency-cruiser evaluates the broader graph topology. The reachable: false attribute identifies "dead wood"—orphaned modules or deprecated infrastructure adapters that are entirely disconnected from the application's primary entry points. Additionally, the numberOfDependentsMoreThan attribute acts as a structural heuristic; it can verify if a shared utility module placed in a common directory is genuinely shared across multiple components, or if it represents an architectural anomaly only used by a single file.Furthermore, dependency-cruiser allows rules to be tightly restricted by dependencyTypes. This is crucial in Node.js environments to ensure that production code located in inner architectural layers does not inadvertently import devDependencies (such as testing frameworks or mock data generators), thus maintaining the purity and security of the production build.Tool Analysis 3: The PHP Ecosystem (Deptrac)In the PHP landscape, Deptrac serves as the premier static code analysis tool for explicitly defining, communicating, and enforcing architectural decisions within large-scale projects. Due to PHP's dynamic, interpreted nature and runtime class loading mechanisms, bytecode analysis is less viable. Instead, Deptrac relies on deeply parsing the raw PHP source code into an Abstract Syntax Tree (typically leveraging robust community libraries like nikic/php-parser) to resolve namespace imports, trace class inheritance hierarchies, and verify interface implementations.Collectors and LayersDeptrac abstracts the raw codebase into manageable architectural concepts using two primary mechanisms: "Layers" and "Collectors". A software architect defines a Hexagonal Architecture setup within a YAML or PHP configuration file located in the project root.Layers represent the abstract architectural boundaries (e.g., Domain, Application, Infrastructure, UI).Collectors are the heuristic engines utilized to intelligently assign physical PHP classes to these logical layers. Deptrac distinguishes itself by supporting a highly flexible array of collector types:ClassName / Directory Collectors: These utilize regular expressions to match fully qualified namespaces or directory paths. For example, a collector can map any class residing under App\Domain\.* directly to the Domain layer.Implements / Extends Collectors: This powerful feature maps any class that implements a specific interface or extends a base class to a designated layer. This enables highly semantic architectural grouping that completely transcends physical file locations, accurately categorizing adapters based on behavior rather than naming.Attribute Collectors: In modern codebases, this leverages PHP 8 attributes (annotations) to explicitly tag classes with their intended architectural layer directly in the source code, eliminating heuristic guesswork.Enforcing Hexagonal RulesetsOnce the codebase is systematically bucketed into layers via the collectors, Deptrac evaluates a strictly defined ruleset against the extracted AST dependency graph. To enforce Hexagonal boundaries, the ruleset defines exact inter-layer permissions. For instance, the configuration will state that the Application layer is permitted to depend on the Domain layer, but the Domain layer's allowed dependency array is left intentionally empty. This configuration explicitly forbids any outbound connections from the core.Designed for continuous enforcement, Deptrac is routinely integrated seamlessly into CI/CD pipelines. During automated checks, it analyzes pull requests, calculates any structural violations against the ruleset, and can output rich visualizations using tools like Graphviz or Mermaidjs to highlight the exact lines of code causing architectural failures, ensuring debt never enters the mainline branch.Tool Analysis 4: The.NET Ecosystem (NDepend)For enterprise applications built on.NET and C#, NDepend provides a uniquely sophisticated static analysis engine. Unlike the previously mentioned tools which primarily operate as CLI utilities, NDepend boasts an incredibly deep integration with the Visual Studio IDE, offering architects real-time feedback via interactive dependency matrices, sprawling dependency graphs, and a proprietary query language.Code Query LINQ (CQLinq)NDepend's most powerful and distinctive feature is CQLinq (Code Query LINQ). This capability empowers developers and architects to query the structure of the compiled.NET codebase using the standard, highly familiar C# LINQ syntax.Architectural rules are expressed as functional queries executed against NDepend's generalized dependency model, which evaluates relationships through the IUsed and IUser interfaces. To statically detect a Hexagonal Architecture violation—such as a scenario where the Domain core inappropriately depends on a database adapter—an architect can author a concise CQLinq query utilizing custom extension methods:C#from m in Methods 
where m.IsUsing("Infrastructure.Database") && m.IsUsedBy("Domain.Core")

This syntax allows for unparalleled granularity in violation detection. It is capable of isolating specific property assignments, internal method calls, or interface implementations that cross forbidden boundaries, returning the exact offending members rather than just failing a file.Critical Rules, Matrices, and CI IntegrationTo enforce governance, NDepend introduces the concept of "Critical CQLinq Rules." By flagging an architectural rule as critical (such as an outright violation of the inward-flowing dependency mandate), the tool automatically fails a Continuous Integration build process if the rule is broken. Within the Visual Studio IDE, NDepend provides immediate, unobtrusive visual indicators—such as red progress circles or flags in the explorer panel—the moment a developer writes code that creates an illegal dependency. This drastically shortens the feedback loop, preventing technical debt from ever reaching version control.Furthermore, NDepend leverages sophisticated Dependency Matrices to untangle legacy code. By plotting classes on the X and Y axes, architects can visually pinpoint exactly which specific properties or methods are causing a layer violation. This single property locks the entire interface to the UI presentation layer, a violation that is immediately obvious in a matrix view but easily missed in thousands of lines of code.The Next Evolution: Synthesizing Semantic Graphs with Tree-SitterWhile tools like ArchUnit, Deptrac, and NDepend are highly effective, they are inherently tethered to their specific language ecosystems, requiring teams in polyglot environments to maintain fragmented, specialized configurations. The future of static architectural analysis—and the explicit goal of replacing simplistic regex-based checkers with universal, genuine semantic understanding—lies in language-agnostic AST parsing. The preeminent technology facilitating this paradigm shift is Tree-sitter.Incremental Parsing and Concrete Syntax TreesTree-sitter is an advanced parser generator tool and incremental parsing library capable of generating highly efficient Concrete Syntax Trees (CSTs) for over 40 programming languages. Written entirely in pure C11, its runtime is dependency-free, making it embeddable in virtually any application.Tree-sitter's primary advantage is its robustness and speed. It is explicitly designed to parse code that contains syntax errors, providing useful structural results even when the code is incomplete or broken. Furthermore, it performs incremental parsing, meaning it can update the syntax tree efficiently on every keystroke within a text editor in milliseconds. By standardizing the structural representation of source code into generic node types (such as class_declaration, method_definition, and import_statement) across different programming languages, Tree-sitter enables the creation of universal architectural rules that apply equally to Java, TypeScript, or Rust.AST Queries via S-ExpressionsTo extract architectural insights from the generated syntax trees, Tree-sitter employs a sophisticated query language based on S-expressions. This system operates conceptually similar to CSS selectors or XPath for XML, allowing developers to target highly specific structural patterns within the code graph.For example, to statically detect if an Adapter class correctly implements a Port interface (thereby satisfying the missing port violation pattern), an analysis engine can execute a Tree-sitter query against Java or TypeScript code:Scheme(class_declaration 
  name: (identifier) @class_name 
  interfaces: (interface_type_list (type_identifier) @interface_name))
This S-expression query precisely isolates the class declaration, binding the specific class name and the implemented interface to variables (@class_name and @interface_name). This approach completely bypasses the extreme fragility of regex string matching. Utilizing Tree-sitter's tag extraction system, the tool can definitively differentiate between a standard object reference (@reference.class) and an interface realization (@reference.implementation), providing the exact semantic depth required for true Hexagonal validation.The Boundary Tool: Applying Tree-Sitter to Domain-Driven DesignThe practical application of Tree-sitter for enforcing Hexagonal Architecture is perfectly exemplified by Boundary, a modern, Rust-based static analysis tool. Boundary utilizes Tree-sitter's core engine to parse source code across Go, Rust, TypeScript, and Java simultaneously, bridging the polyglot gap.Boundary parses the AST to extract a rich inventory of software components: tracking interfaces, structs, method signatures, domain events, and value objects. It maps files to logical layers utilizing a .boundary.toml configuration file, and subsequently constructs an exhaustive, cross-file dependency graph.Crucially, Boundary is specifically engineered to target Hexagonal Architecture and Domain-Driven Design (DDD) principles out of the box. As modern software systems grow in complexity, and as the deployment of autonomous LLM coding agents accelerates the potential for architectural erosion, the topological isolation of the core domain from infrastructural dependencies has never been more critical.The industry currently relies on powerful, language-specific tools to bridge the gap between conceptual architecture and physical implementation. ArchUnit leverages the ASM bytecode library to trace dependencies with perfect compilation accuracy in Java, ensuring that even deeply nested method calls do not violate layer boundaries. Dependency-Cruiser tackles the dynamic nature of JavaScript by constructing massive import graphs navigated via regex path heuristics. By converting raw source code into standardized Concrete Syntax Trees, tools built upon this foundation empower architects to write unified S-expression queries. These modern engines are capable of autonomously detecting missing ports, identifying the illicit injection of concrete adapters, and tracing circular layer dependencies across multiple languages simultaneously. Transitioning from lexical regex validation to authentic semantic graph analysis represents the definitive mechanism for preventing architectural drift, maintaining testability, and ensuring the long-term viability of software systems.

Static Analysis of SOLID Principles: Architectural Mechanisms and Thresholds within SonarQubeThe evaluation of software architecture through static analysis requires translating abstract design philosophies into deterministic, mathematically verifiable heuristics. The SOLID principles—Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, and Dependency Inversion—represent a foundational framework for object-oriented design, guiding developers toward maintainable, scalable, and robust systems. However, these principles are inherently semantic and contextual. A static analysis engine, operating strictly on textual representations of code, cannot natively comprehend conceptual concepts like "responsibility" or "abstraction." Instead, production-grade tools like SonarQube operate on syntactic structures, specifically Abstract Syntax Trees (AST) and Control Flow Graphs (CFG), to infer architectural intent.This exhaustive report investigates how enterprise-grade static analysis tooling, specifically SonarQube, detects violations of the SOLID principles across multiple programming languages, with a primary focus on Java and C#. It details the precise rule identifiers, the empirical numeric thresholds established by SonarSource, the specific AST nodes queried during analysis, and the architectural differences between Java's custom AST parsing and C#'s Roslyn-based analyzers. For engineering teams building customized SOLID detectors, understanding these production-grade implementations is critical to establishing baselines that minimize false positives while effectively capturing architectural decay.1. The Epistemology of Static Analysis in Architectural VerificationSonarQube categorizes its static analysis rules into specific software qualities: Maintainability, Reliability, and Security. Violations of the SOLID principles primarily fall under the Maintainability domain, classified technically as "Code Smells". SonarSource operates on a highly stringent philosophy for maintainability rules: the operational target is zero false positives. This philosophy dictates that the static analyzer must err on the side of caution; it is better to miss a nuanced architectural violation than to overwhelm developers with non-actionable alerts, which inevitably leads to alert fatigue and the abandonment of the tool.Because SOLID principles are abstract, SonarQube relies on proxy metrics. By measuring structural complexity, dimensional size, coupling coefficients, and specific syntactic anti-patterns, the engine infers when a class or method has likely breached an architectural boundary. The tool does not compute classic object-oriented metrics in isolation to report them; instead, it introduces the idea of those measures directly into targeted, actionable rules.1.1 Parsing Engines: Sonar-Java versus RoslynTo comprehend how SonarQube queries AST nodes for SOLID violations, it is necessary to examine the underlying parsing engines, as the architecture differs significantly between languages.For Java, SonarQube utilizes a custom parser housed within the sonar-java open-source repository. This parser constructs an AST and traverses it using the Visitor design pattern. The analyzer visits specific node types—such as ClassTree, MethodTree, VariableTree, and MethodInvocationTree—and aggregates state during the traversal. Furthermore, the Java analyzer builds a robust semantic model alongside the AST, allowing rules to resolve types, trace variable declarations, and understand the inheritance hierarchy across the entire compilation unit.For C# and VB.NET, the architecture is fundamentally different and heavily leverages Microsoft's ecosystem. SonarQube relies on the Roslyn compiler framework (sonar-dotnet). Historically, C# analysis was performed using FxCop, but Microsoft deprecated FxCop in favor of Roslyn analyzers. The csharpsquid ruleset implemented by SonarQube for C# effectively acts as a wrapper and executor for Roslyn analyzers. This means that C# rules execute natively during the MSBuild phase, querying Roslyn's SyntaxNode structures and its ISymbol semantic models. Issues are then imported into the SonarQube server as generic reports or natively through the SonarScanner for MSBuild. Consequently, while the conceptual SOLID rules remain consistent across Java and C#, the underlying implementation relies on entirely different AST representations and compiler APIs.2. The Single Responsibility Principle (SRP) Detection MechanismsThe Single Responsibility Principle (SRP) posits that a class should have only one reason to change, meaning it should encapsulate a single, highly cohesive concept or business capability. Because "responsibility" cannot be quantified lexically, SonarQube enforces SRP through a confluence of dimensional limits, coupling constraints, and cognitive load measurements. When these metrics exceed empirically validated thresholds, the analyzer assumes the class or method has taken on too many roles.2.1 Class Coupling Limitations (Rule S1200)The most direct proxy for a lack of single responsibility is excessive coupling. If a class interacts with a vast number of other distinct classes, it is statistically probable that it is coordinating multiple, disparate responsibilities, acting as an orchestrator rather than a cohesive unit. SonarQube implements this detection mechanism via Rule java:S1200 and csharpsquid:S1200: Classes should not be coupled to too many other classes (Single Responsibility Principle).The default numeric threshold for S1200 is empirically set to exactly 20 distinct classes. If a class references more than 20 distinct types, the rule triggers an issue, flagging the class declaration.In the open-source sonar-java repository, the implementation is housed within the file path java-checks/src/main/java/org/sonar/java/checks/ClassCouplingCheck.java. The AST visitor traverses the ClassTree node and intercepts every type declaration, variable initialization, method parameter type, return type, and thrown exception within the class boundaries. As it encounters these references, it resolves their semantic types and constructs a Set<Type> to track unique dependencies. Crucially, the rule is highly aggressive and literal: it counts standard library classes, such as java.lang.Integer, java.lang.String, and java.util.List, toward the limit of 20. Therefore, if a developer utilizes the Integer class, they have consumed 5% (1/20) of the class's coupling budget.While this may seem overly restrictive, it intentionally forces developers to recognize that heavy reliance on primitive wrappers and varied standard data structures often indicates an algorithmic "God Class" that should be decomposed. Engineering teams often configure quality profiles to increase this threshold if their specific framework inherently requires high boilerplate coupling, but the default remains 20 to enforce strict architectural hygiene.Rule IDNamePrimary AST Target NodeDefault ThresholdMetric UnitS1200Classes should not be coupled to too many other classesClassTree20Distinct Referenced TypesS1448Classes should not have too many methodsClassTree35Declared MethodsS1541Method complexity should not be too highMethodTree10Branching NodesS3776Cognitive Complexity of methods should not be too highMethodTree15Cognitive Increment ScoreS1996Files should contain only one top-level classCompilationUnitTree1Top-level Class/Interface2.2 Dimensional Capacity and the "God Class" (Rule S1448)A secondary, yet equally critical, indicator of an SRP violation is raw dimensional size. SonarQube flags the classic "God Class" anti-pattern utilizing Rule java:S1448 and its equivalents: Classes should not have too many methods. A class that grows infinitely in terms of defined behaviors tends to aggregate unrelated responsibilities, becoming a central point of failure and a maintenance bottleneck.The default limit for this rule is 35 methods, defined by the maximumMethodThreshold parameter. When analyzing the AST, the parser queries the ClassTree nodes and iterates over their members() collection. It filters specifically for nodes of type MethodTree. To ensure fairness and accuracy, the rule excludes synthetic methods generated by the compiler, standard constructors, and simple property accessors (getters and setters), as these do not meaningfully contribute to the class's core responsibility burden.Furthermore, the rule includes a configuration parameter named countNonpublicMethods, which is set to true by default. This is a vital architectural decision by SonarSource: it ensures that utility classes attempting to hide massive complexity behind private access modifiers are still evaluated accurately for their internal burden. If the aggregated count of standard and private logic methods exceeds 35, the code smell is reported directly on the class declaration line, advising an immediate refactoring to split the entity.2.3 File Granularity and Structural Cohesion (Rule S1996)To prevent the physical amalgamation of unrelated entities, SonarQube enforces Rule java:S1996: Files should contain only one top-level class or interface each. By querying the CompilationUnitTree (which represents the physical file), the engine counts the number of ClassTree, InterfaceTree, or EnumTree declarations that exist at the absolute root level of the file's syntax tree.If the count is greater than one, it violates the structural cohesion expected of a single-responsibility module. A file that grows to contain multiple top-level classes aggregates too many responsibilities and inevitably becomes harder to understand and maintain. Interestingly, this specific rule is not explicitly ported to the C# analyzer (csharpsquid) out of the box. This omission exists because the.NET ecosystem already relies heavily on Microsoft's StyleCop analyzer, which enforces rule SA1402 to achieve the exact same restriction, rendering a duplicate SonarQube rule redundant.2.4 Complexity Metrics: Cyclomatic versus Cognitive (Rules S1541 and S3776)When classes are properly segregated into modular components, their constituent methods must also reflect single responsibilities. Method-level SRP violations are evaluated using advanced complexity algorithms that map to human comprehension.Historically, Cyclomatic Complexity, defined by Thomas McCabe, was the industry standard. SonarQube implements this via Rule java:S1541 and csharpsquid:S1541. It triggers when a method's complexity exceeds a predetermined limit, often defaulting to 10. The algorithm operates by analyzing the CFG or the MethodTree AST node. It increments the complexity score by one for every branching node encountered, including if, while, for, case, and catch statements. While mathematically sound, Cyclomatic Complexity often fails to accurately represent maintainability, as a massive switch statement with 20 simple cases scores a 20, whereas deeply nested, convoluted logic might score a 5.To address this, SonarSource pioneered Cognitive Complexity, implemented via Rule java:S3776 and csharpsquid:S3776: Cognitive Complexity of methods should not be too high. Cognitive Complexity was designed specifically to better approximate human understanding and true responsibility evaluation.The default threshold for S3776 is set to 15. Unlike Cyclomatic Complexity, Cognitive Complexity penalizes deep nesting. If an IfStatement is found at the top level of a method, it increments the score by 1. However, if an IfStatement is nested within a ForStatement, the inner if increments the score by 2. A third level of nesting increments by 3, and so forth. Conversely, a flat switch statement, regardless of the number of cases, is treated much more leniently because it is cognitively easier to read.This algorithm inherently encourages adherence to the Single Responsibility Principle. The only practical way to reduce a Cognitive Complexity score below the threshold of 15 is to extract complex, nested logic into independent sub-methods. By breaking down "God Methods" into smaller, specialized functions, the code naturally aligns with SRP guidelines.3. The Open/Closed Principle (OCP) HeuristicsThe Open/Closed Principle dictates that software entities—classes, modules, and functions—should be open for extension but closed for modification. In practical software engineering, this means architectures should avoid monolithic decision trees that must be manually updated every time a new entity, type, or behavior is introduced into the system. Static analysis tools detect OCP violations by identifying syntactic anti-patterns that represent hardcoded, brittle conditional structures rather than dynamic, polymorphic dispatch.3.1 Polymorphic Deficiencies and instanceof CascadesA primary, universally recognized indicator of an OCP violation is the repeated and cascaded use of type-checking operators. When developers write cascading logic blocks such as if (obj instanceof TypeA) {... } else if (obj instanceof TypeB) {... }, they are failing to leverage the fundamental object-oriented concept of polymorphism. If a new type, TypeC, is introduced to the application, the core logic containing this cascade must be physically modified to accommodate the new type—a direct and explicit violation of being "closed for modification".While SonarQube identifies these structures via the AST (by looking for InstanceOfTree nodes in Java or IsPatternExpressionSyntax in C#), it largely evaluates and penalizes them through its branching complexity limits rather than a singular, dedicated "anti-instanceof" rule. Massive cascades of instanceof inherently trigger the Cognitive Complexity (S3776) and Cyclomatic Complexity (S1541) rules due to the repeated if/else branching.The underlying philosophy is that the semantic models of the application should dictate behavior. Instead of querying an object for its type to execute external logic, the architecture should define an interface method (e.g., calculateArea()), implement it across TypeA and TypeB, and allow polymorphic dispatch to handle the execution natively. When static analyzers encounter deep type-switching, it strongly suggests a procedural design masquerading as an object-oriented one.3.2 Switch Statement Constraints (Rules S131, S1301, S1479)Switch statements, particularly when switching over object types, enumerations, or string constants, are mathematically and architecturally identical to instanceof cascades in their OCP implications. Consequently, SonarQube heavily regulates switch structures through multiple interlocking rules.Rule S1479 (Switch with too many cases): The AST query locates SwitchStatement or SwitchExpression nodes. It traverses the children to count the total number of CaseLabelTree nodes. If the number of cases breaches the configured threshold, it triggers an alert. This metric operates on the assumption that a massive switch block is a centralized control structure that violates OCP; introducing a Factory Pattern, a Strategy Pattern, or standard polymorphism should replace the monolithic control mechanism.Rule S1301 (Switch at least three cases): If a switch block contains fewer than three cases, it should be refactored into an if-else statement. This enforces correct and minimal control flow selection, ensuring that heavy switch constructs are not used unnecessarily.Rule S131 (Switch statements should have default clauses): To ensure safety against unknown future extensions, every switch must explicitly define a default path. If the AST visitor determines that a SwitchStatement lacks a default case, it flags the block as a blocker-level code smell. This ensures that when the system is extended, unexpected inputs are handled gracefully without modifying the existing switch body.3.3 Deliberate Exceptions for Extension (Rule S1172)To satisfy the "open for extension" requirement of OCP, developers frequently create abstract base classes or generic interfaces containing methods that are explicitly meant to be overridden by future client code. This architectural design creates an interesting conflict with standard static analysis hygiene rules, specifically those looking for dead code.SonarQube's Rule java:S1172 (Unused method parameters should be removed) contains highly specific algorithmic bypasses precisely to accommodate the Open/Closed Principle. Normally, if a parameter is passed in a method signature but its IdentifierTree is never referenced within the MethodTree body (e.g., in a MethodInvocation or assignment), the analyzer flags it as useless dead code.However, the rule's implementation explicitly ignores unused parameters if the AST satisfies any of the following OCP-related conditions:Intentional Extension Hooks: The method body contains a ThrowStatement that exclusively throws an UnsupportedOperationException. This is a standard Java pattern indicating that the current class provides a default empty interface, and subclasses are expected to implement the logic and utilize the parameters.Overridable Signatures: The method is designed to be overridden. The AST visitor checks if the method body is completely empty and non-private, or if the method possesses an @Override annotation, indicating it is fulfilling an external contract where it cannot change the signature even if it doesn't need the parameter.This demonstrates a highly nuanced implementation where the AST visitor actively looks for specific block structures and annotations to suppress warnings that would otherwise conflict with standard OCP implementation patterns. By recognizing these specific syntactic fingerprints, SonarQube ensures it does not penalize architectures designed for future extension.4. The Liskov Substitution Principle (LSP) AnalysisThe Liskov Substitution Principle, introduced by Barbara Liskov, represents the mathematical definition of behavioral subtyping. It mandates that objects of a superclass should be replaceable with objects of a subclass without affecting the correctness or the established contracts of the program. The formal definition requires strict adherence to invariants: preconditions cannot be strengthened in a subtype, and postconditions cannot be weakened.Static analysis tools inherently lack dynamic runtime context. Without executing the code, maintaining memory state, or utilizing advanced formal proof solvers, true behavioral verification of LSP is impossible. Consequently, SonarQube approximates LSP validation by enforcing syntactic contract strictness through annotation inspection and rigid type hierarchy analysis.4.1 Contract Tightening via Nullability AnnotationsSonarQube analyzes subclass parameter overriding using a specific algorithmic rule designed entirely to prevent contract tightening. If a parent class defines a method contract that explicitly accepts null values as valid input, a subclass overriding that exact method cannot suddenly reject null values. Doing so would break any polymorphic client code that relies on the parent's looser contract, leading to runtime crashes and violating LSP.To detect this without execution, the analyzer relies entirely on semantic symbol resolution and AST annotation nodes. The process operates as follows:The AST parser locates a MethodTree declaration that overrides a parent method.It queries the semantic model (ISymbol in Roslyn or the semantic type API in Java) to locate the exact parent method's signature in the inheritance tree.It extracts metadata annotations specifically related to nullability from both the parent and the child method parameters. It looks for known annotations such as @Nullable, @CheckForNull, @NotNull, @NonNull, and @Nonnull.If the semantic comparison reveals that the parent method's parameter is annotated with @Nullable, but the overriding subclass method annotates the same parameter with @NotNull, the analyzer immediately flags this as an LSP violation. SonarQube relies on semantic symbol resolution to compare method overriding signatures. An LSP violation is flagged when a subclass illegally strengthens a parameter's precondition (e.g., from @Nullable to @NotNull) or weakens a return type's postcondition. Similarly, if the parent method guarantees a non-null return value (@NotNull postcondition), the subclass cannot loosen the contract by suddenly returning @Nullable. This logic relies heavily on the presence of these annotations; if the code is unannotated, static analysis is largely blind to these specific LSP violations.4.2 Type Overrides, Exceptions, and Structural Integrity (Rule S2134)Beyond nullability, another avenue for LSP violation is the modification of expected runtime behavior that breaks framework or platform contracts. SonarQube enforces behavioral consistency in highly specific scenarios using targeted rules.For example, Rule java:S2134 (ThreadOverridesRunCheck) ensures that when developers interact with standard concurrency frameworks, they do not violate the expected operational contracts. If a class extends java.lang.Thread, it is fundamentally expected to override the run() method to execute parallel logic. If a developer subclasses Thread but calls methods that break the concurrent contract or fails to utilize the subclass correctly, the analyzer flags it as a structural breach.Furthermore, static analysis tools implicitly enforce LSP regarding exception handling. In Java, the compiler itself prevents a subclass from introducing new checked exceptions that the parent class did not declare in its throws clause. However, static analyzers extend this by flagging unchecked exception violations or improper casts where standard expected contracts are inexplicably broken, such as detecting when a subclass relies on an unsupported operation that the parent type generally guarantees. While not a catch-all for behavioral defects, these syntactic safety nets form the baseline of automated LSP enforcement.Principle TargetDetection MechanismSpecific AST Nodes / Metadata QueriedOCP (Extensions)Parameter Usage ExceptionsMethodTree, ThrowStatement, @Override annotationsOCP (Modifications)Control Flow MonolithsSwitchStatement, CaseLabelTree, InstanceOfTreeLSP (Preconditions)Contract TighteningMethodTree inheritance, @Nullable, @NotNull annotationsLSP (Behavior)Expected Framework OverridesClass inheritance trees (e.g., extending Thread)5. The Interface Segregation Principle (ISP) ProxiesThe Interface Segregation Principle (ISP) specifies that clients should not be forced to depend upon interfaces that they do not use. This principle directly addresses the "fat interface" anti-pattern—interfaces bloated with multiple, disjointed method declarations that serve disparate client needs.5.1 The Limitations of Static ISP DetectionDetecting an ISP violation via static analysis encounters a fundamental logical limitation: an interface is only technically "fat" relative to the specific client consuming it. If an interface possesses 20 methods, and a single client class utilizes all 20 methods to perform a cohesive task, no ISP violation exists. However, if a different client implements or consumes that same interface but only utilizes 2 of the 20 methods, leaving the other 18 as empty implementations or ignored contracts, the interface is poorly segregated for that specific use case.Because tracking exhaustive cross-project client usage is often computationally expensive and sometimes invisible to a localized AST parser (especially if the interface is part of a published library consumed by external downstream systems), SonarQube tackles ISP primarily through broad dimensional limits rather than complex usage graph analysis.5.2 Measuring Interface Size via Dimensional HeuristicsTo detect bloated API contracts, SonarQube repurposes its class-level dimensional heuristics for interface evaluation. Rule java:S1448 (Classes should not have too many methods) is the primary algorithmic mechanism leveraged against interfaces.When the parser visits an InterfaceTree node, it aggregates the total number of method declarations contained within it. If this count exceeds the established threshold of 35 methods, the analyzer concludes that it is practically impossible for the interface to be fully utilized without burdening the implementing classes with unnecessary dependencies.While the default threshold is configured to 35 for both classes and interfaces, software architecture teams frequently customize their SonarQube Quality Profiles to lower this threshold specifically for interfaces. Dropping the interface threshold to 10 or 15 strictly enforces ISP, compelling developers to divide fat interfaces into modular, highly granular components. For instance, segregating a massive monolithic IMultiFunctionDevice interface into tightly focused IPrinter, IScanner, and IFax interfaces dramatically reduces downstream coupling and perfectly satisfies the static analysis heuristic. By enforcing dimensional limits, SonarQube ensures that interfaces remain focused, even without executing a full global usage analysis.6. The Dependency Inversion Principle (DIP) EnforcementThe Dependency Inversion Principle demands that high-level operational modules should not depend on low-level detail modules; rather, both should depend on abstractions (such as interfaces or abstract classes). Furthermore, abstractions should not depend on details; details should depend on abstractions. In standard programming syntax, violating DIP typically involves the direct instantiation of concrete classes via the new keyword within core business logic, permanently binding the high-level policy to a specific implementation.6.1 Direct Instantiation and Coupling LimitsDetecting the direct instantiation of a class is syntactically trivial for a static analyzer. The AST parser simply locates NewClassTree nodes in Java or ObjectCreationExpressionSyntax nodes in C# and resolves the target semantic symbol. If the target symbol resolves to a concrete class rather than an interface, struct, or primitive, it represents a concrete dependency.However, automatically flagging every new keyword would generate massive and debilitating false positives, completely violating SonarSource's core product philosophy. Developers must instantiate data transfer objects (DTOs), collections, and domain entities constantly. Therefore, SonarQube does not possess a blunt "do not use new" rule.Instead, SonarQube addresses DIP at a broader architectural scale by utilizing the S1200 Class Coupling rule as a proxy. By strictly limiting total class dependencies to 20, the engine inherently forces developers to inject abstract interfaces rather than directly instantiating dozens of concrete services. If a business logic class requires 25 distinct concrete utility classes to operate, the resulting S1200 rule violation acts as a primary, macroscopic indicator of a DIP breach. To pass the check, the developer must abstract those utilities behind a few highly cohesive interfaces and inject them, satisfying both SRP and DIP simultaneously.6.2 Framework-Specific Inversion of Control (IoC) RulesBecause Dependency Injection (DI) is primarily handled by external, domain-specific frameworks (like Spring in Java or the Microsoft.Extensions.DependencyInjection library in.NET), SonarQube employs highly targeted rules that parse framework-specific annotations to enforce DIP.In Java Spring applications, dependency inversion is managed via the IoC container. SonarQube ensures adherence to robust DIP architectures through several specialized rules:Rule S7178 (StaticFieldInjectionNotSupportedCheck): Injecting dependencies into static fields breaks object-oriented polymorphism and lifecycle management, severely undermining DIP. The AST parser queries variable declarations with the static modifier that are also tagged with @Autowired or @Inject. If found, it flags a severe violation.Rule S3305 (SpringConfigurationWithAutowiredFieldsCheck): Field injection is generally discouraged in enterprise architectures in favor of constructor injection. Constructor injection mathematically guarantees that the class possesses its required dependencies upon creation, ensuring a robust and verifiable DIP state. This rule alerts developers when they rely on reflection-based field injection rather than immutable constructors.Rule S4602 (SpringScanDefaultPackageCheck): Placing core application annotations like @SpringBootApplication or @ComponentScan in the default, unnamed package forces the framework to blindly scan every single class in the application's classpath. This destroys module isolation and leads to untracked, bloated dependency graphs. The parser analyzes the PackageTree node; if it evaluates to empty or null while the annotation is present, the architecture is flagged for a structural violation.6.3 Design Patterns and the Singleton Anti-Pattern (Rule S6548)The Singleton design pattern is arguably the antithesis of the Dependency Inversion Principle. It creates an opaque, globally accessible, concrete dependency that fundamentally resists mocking, testing, and abstraction. SonarQube actively monitors the codebase for this anti-pattern via Rule java:S6548: The Singleton design pattern should be used with care.The detection mechanism for this rule is a prime example of complex AST querying. The static analyzer looks for a highly specific architectural fingerprint by querying ClassTree nodes for the simultaneous presence of three structural conditions:A field declaration possessing both private and static modifiers that holds an instance of the class itself.A constructor possessing a private access modifier, which physically prevents external, flexible instantiation.A method possessing public and static modifiers (conventionally named getInstance()) that returns the private instance.When this exact structural triad is detected within a single class boundary, the engine raises a design-level maintainability smell. This alert encourages developers to dismantle the rigid singleton and instead refactor the architecture to utilize proper Dependency Injection, thereby restoring alignment with the Dependency Inversion Principle.7. Open Source Implementations and ExtensibilityFor engineering teams seeking to build customized SOLID detectors or deeply integrate with existing CI/CD pipelines, understanding the open-source nature and extensibility of these production-grade rules is paramount.7.1 GitHub Repositories and File PathsThe rules governing Java are open source and actively maintained by SonarSource on GitHub. The implementation logic for the custom Java AST parser and its associated SOLID checks can be found within the SonarSource/sonar-java repository. For example, the precise logic that calculates the coupling metrics and enforces the 20-class limit is located at the file path java-checks/src/main/java/org/sonar/java/checks/ClassCouplingCheck.java. Other open-source contributions to the SonarQube ecosystem, such as Refaster rules used in Error Prone, can be found in community repositories like tech/picnic/errorprone/refasterrules/TestNGToAssertJRules.java.For the.NET ecosystem, the C# and VB.NET analyzers are hosted in the SonarSource/sonar-dotnet repository. Because these rely on Microsoft's Roslyn compiler API, the implementation details differ wildly from Java. The source code for complex parsing logic, such as lexical analysis and syntax tree traversal, can be explored in files like sonaranalyzer-dotnet:analyzers/src/SonarAnalyzer.Common/Json/Parsing/LexicalAnalyzer.cs and sonaranalyzer-dotnet:analyzers/src/SonarAnalyzer.VisualBasic/Facade/VisualBasicSyntaxFacade.cs.7.2 Customizing and Creating RulesSonarQube’s default thresholds, such as 35 methods for S1448 or a Cognitive Complexity of 15 for S3776, are empirically validated against hundreds of massive open-source repositories to minimize false positives globally. However, organizational standards often vary based on domain constraints and architectural maturity.SonarQube accommodates this variance by allowing users to modify the parameters of these rules dynamically via "Quality Profiles" within the administrative UI. A software architecture team could easily redefine the S1200 coupling limit from 20 down to 10 to mandate highly granular microservices, or reduce the Cognitive Complexity threshold to 10 for safety-critical systems.Furthermore, if internal, domain-specific constraints are required that go beyond the built-in SOLID proxy metrics, both ecosystems offer extensibility. In C#, tools like the Roslyn framework allow developers to write entirely custom DiagnosticAnalyzer classes. These custom analyzers query the Roslyn semantic model directly, identify proprietary anti-patterns, and output standard diagnostics possessing unique CheckIds. These custom diagnostics are then natively ingested and displayed by the SonarScanner during the MSBuild phase.In Java, creating custom checks involves utilizing the sonar-java API. Developers write custom plugins that extend the IssuableSubscriptionVisitor class. By registering the specific AST node types (e.g., Tree.Kind.METHOD or Tree.Kind.CLASS) the custom rule wishes to subscribe to, the visitor pattern will automatically invoke the custom traversal logic whenever the parser encounters those structures during the build. This allows enterprise teams to seamlessly inject proprietary architectural validation into the global SonarQube reporting matrix.8. ConclusionsThe translation of abstract, semantic SOLID principles into statically analyzable, syntactic code metrics requires mapping human architectural intent onto rigid Abstract Syntax Trees. SonarQube achieves this monumental task not by attempting to understand the code, but through a sophisticated, empirically validated matrix of proxy heuristics and dimensional constraints:Single Responsibility (SRP) is strictly monitored by limiting dimensional space (S1448 limits classes to 35 methods, S1996 restricts files to 1 top-level class) and measuring structural connections (S1200 restricts class coupling to a maximum of 20 distinct types). The Cognitive Complexity metric (S3776, limited to 15) acts as the definitive proxy for internal method responsibility bounds by aggressively penalizing nested logic.Open/Closed (OCP) relies heavily on control flow limitations to deter brittle architecture. Monolithic switch statements (S1479) and missing default clauses (S131) denote rigid logic paths unable to be extended without physical modification. Type-checking cascades (instanceof) are penalized via complexity metrics.Liskov Substitution (LSP) pushes static analysis to its absolute limits, requiring semantic verification of inheritance trees to detect illegal contract tightening through the inspection of nullability annotations (e.g., overriding a @Nullable parameter with a @NotNull requirement).Interface Segregation (ISP) leverages class-level dimensional limits (S1448) against interfaces to deter structural bloat, mathematically forcing developers to segregate large API contracts into cohesive subsets.Dependency Inversion (DIP) is primarily protected at a macro level by the S1200 coupling threshold, which necessitates abstract injection to avoid triggering alerts. This is supplemented by rigorous framework-specific rules (S7178, S3305) that police the structural integrity of Inversion of Control implementations and flag anti-patterns like singletons (S6548).Production-grade static analysis tooling demonstrates that while it remains computationally impossible to prove semantic intent mathematically without runtime execution data, a well-calibrated matrix of AST node querying, coupling constraints, annotation inspection, and cognitive complexity thresholds can highly accurately detect structural decay and identify architectural SOLID violations before they reach production.

# Plan 04 — Ratings Presentation, Context Profiles, and Exception Governance

**Priority: P2** — turns the Plan 01 engine's numbers into output developers trust; adds the context-awareness (test/generated code) without which every rating denominator is wrong.
**Research basis:** `_Docs/Research/Bragi/Completed/DEEPTHINK_02` (channel-specific precision/recall operating points, facts vs inquiries), `DEEPTHINK_14` (epistemic honesty, confidence-graded output), `DEEPTHINK_12` (test code: DAMP not DRY, dedicated profile), `DEEPTHINK_03` §5 (generated-code exclusion pipeline), `DEEPTHINK_11` (suppression with justification, baseline ratchet, two-tier config), `DEEPTHINK_07` (severity as empirical risk, not dogma).

---

## 1. Rationale (what the research says)

- DEEPTHINK_02: the acceptable false-positive rate is a function of the **channel** — blocking CI gate ~99% precision, IDE ~80–90%, PR advisory ~70–80%, trend dashboard ~50% with max recall. One undifferentiated findings list guarantees either a muted tool or missed rot. Quantitative findings ("method is 65 lines, project limit 50") must be presented as *facts with a policy*; heuristic findings (Feature Envy-class) as *low-confidence inquiries* — "by acknowledging its own uncertainty, the tool aligns with the developer's mental model".
- DEEPTHINK_12: test code optimizes for DAMP; rating it with production thresholds produces false debt and drives metric-gaming (brittle base-class test hierarchies). Cyclomatic thresholds must *relax* (≈25), cognitive must *tighten* (≈2), duplication clone-size triples, test cases are docstring-exempt while test *infrastructure* (fixtures/factories) is doc-mandatory.
- DEEPTHINK_03 §5: generated code in maintainability metrics is a category error; exclusion must be a zero-trust pipeline (path conventions → 500-byte header peek for `Code generated by`/`@generated`/`DO NOT EDIT` → structural-entropy fallback flagging `SUSPECTED_GENERATED`), while security scanning still runs.
- DEEPTHINK_11: exceptions need **asymmetric friction** — inline suppressions mandatory-reason'd and rejected bare; baselines are a **one-way ratchet** (read-only in CI, auto-shrink on fix, never grow); config restricted to global + directory tiers; AST-fingerprinted (line-number-agnostic) violation identity.
- DEEPTHINK_14/07: ratings must carry *why* and *what it costs* — Plan 01 produces `ScoreCap.reason`, `ScoreConfidence`, `ROIAction`; this plan is where they surface.

## 2. Current state (gap)

- `Ratings/services/ratings_calculator.py` output is four bare letters + counts; `issues` list on `DimensionRating` holds unranked strings. No rationale ("capped by X"), no confidence, no channel awareness.
- No test/generated/script context anywhere in Bragi's rating or debt paths: `Asgard_Test`-style fixtures, generated files, and `__main__` scripts are all scored as production code. `DependencyConfig`/`RequirementsConfig` have ad-hoc `include_tests`/`exclude_patterns`, but Quality/Ratings do not share a context classifier.
- No suppression mechanism at all in Bragi models — findings cannot be acknowledged, so legacy noise is permanent, which DEEPTHINK_11 identifies as the normalization-of-deviance driver.
- `Asgard/common/baseline` exists at package level but Bragi reports don't use AST-stable fingerprints, so any baseline is line-shift-fragile.

## 3. Target state

### 3.1 Context classifier (`Bragi/common` level: `context_classifier.py`, new)

One shared function used by Quality scanners, debt analyzer, Ratings, and Dependencies:

```python
class CodeContext(str, Enum):
    PRODUCTION = "production"
    TEST = "test"                    # path/file/AST signature cascade (DEEPTHINK_12 §2)
    GENERATED = "generated"          # 3-layer funnel (DEEPTHINK_03 §5)
    SUSPECTED_GENERATED = "suspected_generated"
    SCRIPT = "script"                # ModuleScope-dominant entry points

def classify(path: Path, head_bytes: bytes, ast_signals: AstSignals) -> CodeContext
```

- Test detection cascade: path patterns (`tests/`, `_test.`, `test_`, `.spec.`) → AST signatures (pytest/unittest imports, `@Test`-style decorators). AST beats path (a helper library under `tests/` with no test signatures is infrastructure, not a test case).
- Generated: default-on path exclusions + 512-byte header regex + entropy fallback (>2,000 lines, repetitive AST, zero comments) emitting `SUSPECTED_GENERATED` with an informational note, per DEEPTHINK_03.

### 3.2 Context profiles in scoring

Plan 01's `CompositeScoreEngine` gains a `profile` parameter resolved from `CodeContext`:

| Rule dimension | Production | Test profile (DEEPTHINK_12 values) |
|---|---|---|
| cyclomatic threshold | 10 | 25 (INFO only) |
| cognitive threshold | 15 | 2 (kept strict — flat AAA) |
| duplication min clone | ~50 tokens | ~150 tokens |
| docstring requirement | exported callables | test cases exempt; fixtures/factories mandatory |
| LOC penalty | on | off |

- `GENERATED` files: excluded from maintainability/comprehensibility utilities and from all LOC denominators (risk profiles, TDR); security-relevant findings still count (DEEPTHINK_03: "must still be subjected to SAST").
- Test-exclusive health metrics (assertion density 1–10, hermeticity, shared-mutable-state without teardown, test-to-prod LOC ratio 0.5–4.0 warnings) are added as a `TestHealth` category that only exists for `TEST` context files and feeds Plan 01's Reliability pillar for the test tree.

### 3.3 Presentation layer (`Ratings/services/_ratings_presenter.py`, new)

Every rendered rating follows Plan 01's progressive disclosure, with DEEPTHINK_02's fact/inquiry split:

- `FindingClass = FACT | HEURISTIC`: severity vocabulary differs — facts render as violations ("65 lines; project limit 50"), heuristics as suggestions with confidence ("Low confidence: consider moving — accesses 4 properties of `UserAccount`, none of own class").
- **Channel presets** (`ChannelProfile`): `ci_gate` (FACT-class, deterministic metrics only), `pr_review` (FACT + HEURISTIC ≥ medium confidence, differential only — Plan 06), `ide` (unobtrusive full set), `dashboard` (max recall, trend deltas). A finding's minimum channel is part of its rule metadata; the same scan serves all four filters.
- Rating rationale strings mandatory: grade + top cap/driver + top-3 ROI actions ("B — held back by duplication (12.4%); best next action: extract `parse_config` clone family, +0.03").

### 3.4 Exception governance (`Bragi/common/suppressions.py` + baseline upgrade)

- Inline suppression grammar: `# bragi: disable=<rule-id> reason=<text>` (reason mandatory; bare directives are themselves a FACT-class finding). Scanners attach `suppressed=True` instead of dropping — dashboard channel still sees them.
- **Unused-suppression detection**: a directive whose rule no longer fires is a finding (state-based expiry, DEEPTHINK_11's alternative to time bombs).
- Baseline ratchet: `.asgard_baseline.json` of violation fingerprints — `fingerprint = hash(rule_id, module_path, enclosing_symbol, normalized_snippet)` (AST-stable, line-agnostic). CI mode treats it read-only; a `--shrink-baseline` mode removes fixed entries; growth requires an explicit, separately-permissioned command. Integrate with the existing `Asgard/common/baseline` utilities rather than duplicating them.
- Config hierarchy hard-limited to two tiers: project root + directory overrides; per-file threshold overrides rejected by the config loader with a pointer to inline suppression (DEEPTHINK_11 §3).
- Suppression telemetry in `ProjectRatings`: counts by rule → the DEEPTHINK_11 "coach vs cop" review signal (a rule carrying a dominant share of suppressions is a miscalibrated rule; feeds Plan 05's threshold review).

## 5. Phased implementation

1. **Phase A — context classifier**: shared classifier + stamping `context` on findings; GENERATED exclusion from denominators (immediately corrects TDR and risk-profile math from Plans 01/02).
2. **Phase B — test profile**: threshold table, TestHealth category, DAMP duplication relaxation.
3. **Phase C — presentation**: FindingClass metadata on existing rules (one-time triage of rule registry), presenter, channel presets; CLI/MCP consumers updated.
4. **Phase D — governance**: suppression grammar + unused detection; fingerprint baseline ratchet; suppression telemetry into `ProjectRatings`.

## 6. Testing considerations

- Classifier: fixture matrix (pytest file under `src/`, helper under `tests/`, protobuf output with and without header, 3,000-line commentless data file) → expected contexts; property: classification is pure (same inputs → same output).
- Profile: identical smell set in prod vs test fixture → test file's grade unaffected by 12 duplicated AAA setups but penalized for an `if/else` in a test body (cognitive=2 rule).
- Presentation: golden-text tests for fact vs heuristic rendering; channel filter monotonicity (ci_gate ⊆ pr_review ⊆ dashboard finding sets).
- Governance: bare `# bragi: disable` yields a finding; fixed violation + stale directive yields unused-suppression finding; baseline fingerprints survive inserting 10 lines above the violation (line-shift test); baseline cannot grow in CI mode (write attempt raises).
- Backward compat: default profile = PRODUCTION and no baseline file → byte-identical `ProjectRatings` for existing tests.
